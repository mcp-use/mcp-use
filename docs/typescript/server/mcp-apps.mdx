MCP Apps is the **official standard** for interactive widgets in the Model Context Protocol ecosystem. With mcp-use, you return **plain React components as JSX** from tool handlers. The framework bundles widgets, registers resources, and emits MCP Apps and ChatGPT Apps SDK metadata automatically.

<Info>
  **mcp-use Advantage**: Return JSX from `server.tool()` and hosts receive **both** MCP Apps and ChatGPT metadata. No separate `resources/` folder or manual `widget: { name }` wiring for the primary pattern.
</Info>

## Why MCP Apps?

MCP Apps is the recommended widget protocol for MCP servers because it:

- **MCP-native**: Built specifically for the Model Context Protocol ecosystem
- **Open Standard**: Ensuring long-term compatibility across the MCP ecosystem
- **Secure**: Double-iframe sandbox architecture with granular CSP control
- **Feature-rich**: JSON-RPC 2.0 communication with full MCP integration
- **Future-proof**: Supported by a growing ecosystem of MCP clients

## Quick Start

Start with the MCP Apps template:

```bash
npx create-mcp-use-app my-mcp-server --template mcp-apps
cd my-mcp-server
npm install
npm run dev
```

This scaffolds a server that uses the **inline JSX** widget pattern out of the box.

## The inline JSX pattern (recommended)

1. Add the JSX pragma so your server file uses the mcp-use JSX runtime:

```tsx
/** @jsxImportSource mcp-use/jsx */
```

2. Import a **plain React component** from anywhere in your project (no `resources/` convention required).

3. Return JSX from the tool handler. Use **`_`-prefixed props** for protocol metadata (`_output`, `_invoking`, `_csp`, and so on).

```tsx
/** @jsxImportSource mcp-use/jsx */
import { MCPServer, text } from "mcp-use/server";
import { z } from "zod";
import WeatherCard from "./components/WeatherCard";

const server = new MCPServer({ name: "demo", version: "1.0.0" });

export const getWeather = server.tool(
  {
    name: "get-weather",
    description: "Weather for a city.",
    schema: z.object({ city: z.string() }),
  },
  async ({ city }) => (
    <WeatherCard
      city={city}
      temperature={22}
      conditions="sunny"
      humidity={45}
      _output={text(`Weather in ${city}: 22°C, sunny`)}
      _invoking="Fetching weather..."
      _invoked="Weather loaded"
      _csp={{ connectDomains: ["https://api.openweathermap.org"] }}
      _prefersBorder={false}
    />
  )
);

await server.listen();
```

Under the hood, the JSX runtime turns this into a structured tool result (the same shape as the internal `widget()` helper) and attaches **inline widget metadata** for `tools/list` and `resources/read`. You do not call `widget()` yourself.

### `_` props convention

| Prop | Purpose |
| --- | --- |
| `_output` | Model-visible summary — use `text()`, `image()`, etc. |
| `_meta` | Extra `_meta` on the tool result (exposed as `metadata` in `useWidget`) |
| `_invoking` / `_invoked` | Status text while the tool runs / after it completes |
| `_csp` | Per-call CSP (merged with component `.config` and server origin) |
| `_prefersBorder` | Border vs. borderless chrome preference |
| `_fileParams` | Which tool arguments are file handles (for upload UIs) |
| `_visibility` | Tool list visibility hints |
| `_closeWidget` | Request closing the widget after the run |
| `_domain` / `_permissions` | Host-specific domain and sandbox permission hints |

See [Content Security Policy](./content-security-policy) for CSP details.

### Component-level config: `defineWidget` and `.config`

Static options that belong to the **component** (not a single call) can live on the component:

- **`defineWidget(Component, config)`** from `mcp-use/react`, or
- **`Component.config = { ... }`** satisfying `ComponentWidgetConfig`

Merge order: **defaults → `.config` → JSX `_` props** on each invocation.

```tsx
import { defineWidget } from "mcp-use/react";

export default defineWidget(
  function WeatherCard(props: Props) {
    return <div>...</div>;
  },
  {
    csp: { connectDomains: ["https://api.example.com"] },
    prefersBorder: true,
  }
);
```

### `ToolRef` and typed `useCallTool`

`server.tool()` returns a **`ToolRef`**. Pass it to `useCallTool(ref)` for **zero-codegen** input/output typing.

```tsx
import { useCallTool } from "mcp-use/react";
import { getWeather } from "../server"; // same tool ref export

const { callTool, data, isPending } = useCallTool(getWeather);
await callTool({ city: "Paris" });
```

For many tools, **`createTypedHooks({ getWeather, ... })`** returns a typed `useCallTool` bound to your map.

### `defineTools()` batch registration

Register multiple tools and get a typed map:

```tsx
export const tools = server.defineTools({
  echo: {
    description: "Echo input.",
    schema: z.object({ message: z.string() }),
    handler: async ({ message }) => text(`Echo: ${message}`),
  },
});
```

### Streaming props with `streamable()`

For incremental UI updates, pass a **`streamable(initial)`** value as a prop. The server pushes updates to the widget; the widget uses **`useStreamableProps()`** to subscribe.

```tsx
import { streamable } from "mcp-use/server";

const analysis = streamable("");
analysis.update((prev) => prev + chunk);
// ...
analysis.done();

return (
  <AnalysisView
    text={input}
    analysis={analysis}
    _output={text("Analysis in progress...")}
  />
);
```

### Conditional returns: widgets vs. text

When the client cannot render MCP Apps, return plain content:

```tsx
async (params, ctx) => {
  if (ctx.client.supportsApps()) {
    return <WeatherCard {...props} _output={text("...")} />;
  }
  return text("Plain text fallback for clients without Apps support.");
}
```

## MCP Apps vs ChatGPT Apps SDK

| Feature             | MCP Apps (Standard)                    | ChatGPT Apps SDK               |
| ------------------- | -------------------------------------- | ------------------------------ |
| **Protocol**        | JSON-RPC 2.0 over `postMessage`        | `window.openai` global API     |
| **MIME Type**       | `text/html;profile=mcp-app`            | `text/html+skybridge`          |
| **Specification**   | MCP Apps (open standard)               | OpenAI proprietary             |
| **Architecture**    | Double-iframe sandbox                  | Single iframe                  |
| **CSP Format**      | camelCase (`connectDomains`)           | snake_case (`connect_domains`) |
| **Client Support**  | MCP Apps clients (Claude, Goose, etc.) | ChatGPT                        |
| **mcp-use Support** | Full support                           | Full support                   |

mcp-use generates metadata for **both** protocols from your JSX and component config.

## Data Flow: What the LLM Sees vs What the Widget Sees

Per the [MCP Apps specification](https://github.com/modelcontextprotocol/ext-apps), a tool result has fields with different visibility:

| Field | LLM sees it? | Widget sees it? | Purpose |
| --- | --- | --- | --- |
| `content` | **Yes** | Yes | Text summary for the model |
| `structuredContent` | **No** | Yes (as `props`) | Rendering data for the widget |
| `_meta` | **No** | Yes (as `metadata`) | Protocol + custom metadata |

Inline JSX sets `structuredContent` from your component props (non-`_` keys) and `content` from `_output`.

### Server: returning a widget (inline JSX)

```tsx
return (
  <ProductSearch
    query={query}
    results={results}
    _output={text(`Found ${results.length} products matching "${query}"`)}
    _meta={{ totalCount: 1000, nextCursor: "abc" }}
  />
);
```

### Widget: reading data

```tsx
const { props, toolInput, isPending, metadata } = useWidget();

// props     = structuredContent (server-computed props)
// toolInput = tool call arguments from the model
// metadata  = _meta (including your _meta from JSX)
```

The generated **inline widget entry** mounts your root component immediately and spreads **only** `structuredContent` as props (plus streamed prop updates — the same object as `useWidget().props`). It does **not** pass `toolInput` or `isPending` as component props; read them from `useWidget()` as above.

## Display Modes

Widgets can request different display modes from the host:

```tsx
const { displayMode, requestDisplayMode } = useWidget();

await requestDisplayMode("fullscreen");
await requestDisplayMode("pip");
await requestDisplayMode("inline");
```

The host may decline the request — always check `displayMode` for the actual state.

## Follow-Up Messages

Widgets can send messages to the conversation on behalf of the user using `sendFollowUpMessage`:

```tsx
const { sendFollowUpMessage } = useWidget();

<button onClick={() => sendFollowUpMessage("Tell me more about mangoes")}>
  Ask the AI
</button>
```

See [useWidget — Streaming tool arguments](/typescript/server/widget-components/usewidget#streaming-tool-arguments) for streaming tool arguments (`partialToolInput` / `isStreaming`).

## Host Context: Locale, Theme, and More

`useWidget` exposes host context:

```tsx
const { theme, locale, timeZone, safeArea, userAgent, maxHeight } = useWidget();
```

Use the Inspector playground to test locales, devices, and themes.

## Widget Code (plain React)

Your widget **UI** is a normal React component. Hooks like `useWidget`, `useCallTool`, and `McpUseProvider` run inside the **generated** widget bundle; you typically import them only in components that render in the iframe.

```tsx
import { useWidget, McpUseProvider } from "mcp-use/react";

export default function WeatherCard({ city, temperature, conditions }: Props) {
  const { isPending } = useWidget();

  if (isPending) return <div>Loading...</div>;

  return (
    <McpUseProvider autoSize>
      <div className="weather-card">
        <h2>{city}</h2>
        <div>{temperature}°C</div>
        <div>{conditions}</div>
      </div>
    </McpUseProvider>
  );
}
```

## Interactive Widgets with Tool Calling

Use **`useCallTool(toolRef)`** with a **`ToolRef`** from `server.tool()` for typed calls:

```tsx
import { useWidget, useCallTool } from "mcp-use/react";
import { fetchData } from "../../server";

const InteractiveWidget = () => {
  const { props, isPending: propsPending } = useWidget();
  const { callTool, data, isPending } = useCallTool(fetchData);

  if (propsPending) return <div>Loading...</div>;

  return (
    <div>
      <button onClick={() => callTool({ id: props.itemId })} disabled={isPending}>
        {isPending ? "Loading..." : "Fetch Data"}
      </button>
      {data && <div>{JSON.stringify(data.structuredContent)}</div>}
    </div>
  );
};
```

See [useCallTool()](/typescript/server/widget-components/usecalltool).

## State Management

| State type | Where it lives | Example |
| --- | --- | --- |
| **Business data** | MCP server | Tasks, search results |
| **UI state** | Widget (`setState` from `useWidget`) | Selected row, filters |
| **Cross-session** | Your backend | Saved preferences |

### UI state (`setState`)

```tsx
const { state, setState } = useWidget<Record<string, unknown>, MyState>();

setState({ favorites: next });
```

ChatGPT persists via `window.openai.setWidgetState`; MCP Apps uses `ui/update-model-context` for model-visible UI state.

## MCP Apps Bridge API

For low-level access, use `getMcpAppsBridge()` from `mcp-use/react`. Most apps only need `useWidget` and `useCallTool`.

| Feature | MCP Apps | MCP UI |
| --- | --- | --- |
