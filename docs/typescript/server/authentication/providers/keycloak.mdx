The Keycloak provider uses Dynamic Client Registration (DCR). MCP clients discover Keycloak's OAuth endpoints via `.well-known` passthrough and register themselves directly with Keycloak. Your server only verifies the resulting tokens.

## Setup

### 1. Run Keycloak

```bash
docker run -p 8080:8080 \
  -e KEYCLOAK_ADMIN=admin \
  -e KEYCLOAK_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:latest start-dev
```

### 2. Create a Realm

1. Open the Keycloak Admin Console at `http://localhost:8080`
2. Click **Add Realm** and set a name (e.g. `my-realm`)

### 3. Create a Client

1. Go to **Clients → Create**
2. Set **Client ID**: `my-mcp-server`
3. Set **Client Protocol**: `openid-connect`
4. Set **Access Type**: `confidential` (server-side) or `public` (SPA)
5. Set **Valid Redirect URIs** and **Web Origins** to your server URL

### 4. Enable Dynamic Client Registration

1. Go to **Realm Settings → Client Registration → Anonymous Access Policies**
2. Enable **Trusted Hosts** or configure anonymous registration to allow MCP clients to self-register

### 5. Environment variables

```bash
MCP_USE_OAUTH_KEYCLOAK_SERVER_URL=http://localhost:8080
MCP_USE_OAUTH_KEYCLOAK_REALM=my-realm
MCP_USE_OAUTH_KEYCLOAK_CLIENT_ID=my-mcp-server  # Optional: used for audience verification
```

## Configuration

```typescript
import { MCPServer, oauthKeycloakProvider } from 'mcp-use/server'

// Zero-config: reads MCP_USE_OAUTH_KEYCLOAK_* environment variables
const server = new MCPServer({
  name: 'my-server',
  version: '1.0.0',
  oauth: oauthKeycloakProvider()
})

await server.listen(3000)
```

Or pass config explicitly:

```typescript
oauth: oauthKeycloakProvider({
  serverUrl: 'https://keycloak.example.com',
  realm: 'my-realm',

  // Optional: used as the JWT audience claim during verification
  clientId: 'my-mcp-server',

  // Disable JWT verification during development only
  verifyJwt: process.env.NODE_ENV === 'production',

  // Override advertised scopes
  // Default: ['openid', 'profile', 'email', 'offline_access', 'roles']
  scopesSupported: ['openid', 'profile', 'email', 'offline_access', 'roles'],
})
```

## Accessing user info in tools

Keycloak tokens include realm roles in `realm_access.roles`. Access them via `ctx.auth.user`:

```typescript
server.tool(
  {
    name: 'get-user-info',
    description: 'Get authenticated user info',
  },
  async (_args, ctx) => ({
    userId: ctx.auth.user.userId,
    email: ctx.auth.user.email,
    name: ctx.auth.user.name,
    username: ctx.auth.user.username,
    roles: ctx.auth.user.roles,
    scopes: ctx.auth.scopes,
  })
)
```

## Role-based access control

Keycloak realm roles are available on `ctx.auth.user.roles`:

```typescript
server.tool(
  {
    name: 'admin-action',
    description: 'Admin-only action',
  },
  async (_args, ctx) => {
    if (!ctx.auth.user.roles?.includes('admin')) {
      return { content: [{ type: 'text', text: 'Forbidden: admin role required' }], isError: true }
    }

    // ... admin logic
    return { content: [{ type: 'text', text: 'Done' }] }
  }
)
```
