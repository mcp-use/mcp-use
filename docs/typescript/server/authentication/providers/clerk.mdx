## Setup

### 1. Create a Clerk Application

1. Go to [Clerk Dashboard](https://dashboard.clerk.com/)
2. Click **Create application**, give it a name, and click **Create**
3. Note your **Domain** — found under **Configure → Domains**
   - Format: `your-app.clerk.accounts.dev`

### 2. Create an OAuth Application

Clerk requires a pre-registered OAuth client for MCP server authentication:

1. In the Clerk Dashboard, go to **Configure → Developers → OAuth Applications**
2. Make sure that dynamic client registration has been toggled on in the OAuth applications Setting.

## Configuration

```typescript
import { MCPServer, oauthClerkProvider } from 'mcp-use/server'

const server = new MCPServer({
  name: 'my-server',
  version: '1.0.0',
  oauth: oauthClerkProvider({
    domain: process.env.MCP_USE_OAUTH_CLERK_DOMAIN!,
  })
})

server.listen(3000)
```

`domain` can also be set via the `MCP_USE_OAUTH_CLERK_DOMAIN` environment variable.

### Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `domain` | `string` | — | Clerk domain (e.g. `my-app.clerk.accounts.dev`) |
| `verifyJwt` | `boolean` | `true` | Verify JWT signatures via JWKS. Disable only for local development. |
| `fetchUserInfo` | `boolean` | `false` | Fetch additional claims (`email`, `name`, `picture`, org info) from Clerk's `/oauth/userinfo` endpoint for JWT tokens. Opaque tokens always fetch userinfo regardless of this setting. |

<Note>
By default, JWT tokens return only the claims present in the JWT payload. Clerk JWTs do not include `email`, `name`, `picture`, or organization fields. If your tools need these claims, enable `fetchUserInfo: true`. Userinfo responses are cached for 60 seconds.
</Note>

```typescript
const server = new MCPServer({
  name: 'my-server',
  version: '1.0.0',
  oauth: oauthClerkProvider({
    domain: process.env.MCP_USE_OAUTH_CLERK_DOMAIN!,
    fetchUserInfo: true, // Fetch email, name, picture, org info from Clerk
  })
})
```

## Access User Info

`mcp-use` automatically populates `ctx.auth.user` so your tools can read authenticated user data directly.

```typescript
server.tool(
  {
    name: 'get-user-info',
    description: 'Get the authenticated user',
  },
  async (_args, ctx) => {
    return object({
      userId:          ctx.auth.user.userId,
      email:           ctx.auth.user.email,
      name:            ctx.auth.user.name,
      picture:         ctx.auth.user.picture,
      org_id:          ctx.auth.user.org_id,
      org_role:        ctx.auth.user.org_role,
      org_permissions: ctx.auth.user.org_permissions,
      permissions:     ctx.auth.permissions,
      scopes:          ctx.auth.scopes,
    })
  }
)
```

If you need fields that Clerk does not expose through its userinfo endpoint, fetch them explicitly from Clerk's Backend API in your own tool.

## Permissions

Clerk permissions are org-scoped via `org_permissions`.

### Check Org Permissions

```typescript
server.tool(
  {
    name: 'delete-document',
    description: 'Delete a document',
  },
  async ({ documentId }, ctx) => {
    const permissions = ctx.auth.user.org_permissions ?? []

    if (!permissions.includes('delete:documents')) {
      return error('Forbidden: delete:documents permission required')
    }

    await db.documents.delete({ id: documentId })
    return text('Document deleted')
  }
)
```

### Check Organization Role

```typescript
server.tool(
  {
    name: 'admin-action',
    description: 'Admin-only action',
  },
  async (_args, ctx) => {
    if (ctx.auth.user.org_role !== 'admin') {
      return error('Forbidden: admin role required')
    }

    // ... admin logic
    return text('Done')
  }
)
```
