This guide walks you through creating an MCP server whose tools return **inline JSX** — plain React components — with dual MCP Apps and ChatGPT metadata.

## What You'll Build

By the end of this guide, you'll have:

- A working MCP server (`MCPServer` + `server.listen()`)
- Tools that return JSX (`<MyWidget ... />`) with `_output`, `_csp`, and related `_` props
- React components that receive data through `useWidget()` inside the widget iframe
- A dev workflow with the Inspector and HMR

## Prerequisites

- Node.js 18+
- TypeScript and React basics
- Optional: [MCP 101](/home/mcp101)

## Step 1: Create Your Project

```bash
npx create-mcp-use-app my-widget-server --template mcp-apps
cd my-widget-server
npm install
```

Typical layout (names may vary by template version):

```
my-widget-server/
├── components/           # React widget UI (any structure you want)
│   └── ProductSearch.tsx
├── public/
├── index.tsx             # Server entry (TSX with @jsxImportSource)
├── package.json
├── tsconfig.json
└── README.md
```

There is **no required `resources/` folder** for the primary pattern. Put components anywhere and import them into your server file.

<Tip>
Use **`/** @jsxImportSource mcp-use/jsx */`** at the top of the server file that returns JSX from tools so the compiler uses the mcp-use JSX runtime.
</Tip>

## Step 2: Tool that returns a widget (inline JSX)

Open your server entry (often `index.tsx`). A widget tool returns a React element; `_output` is what the model sees.

```tsx
/** @jsxImportSource mcp-use/jsx */
import { MCPServer, text } from "mcp-use/server";
import { z } from "zod";
import ProductSearch from "./components/ProductSearch";

const server = new MCPServer({
  name: "my-widget-server",
  version: "1.0.0",
  baseUrl: process.env.MCP_URL || "http://localhost:3000",
});

server.tool(
  {
    name: "search-tools",
    description: "Search fruits and show results in a widget",
    schema: z.object({
      query: z.string().optional().describe("Filter fruits"),
    }),
  },
  async ({ query }) => {
    const results = fruits.filter(
      (f) => !query || f.fruit.toLowerCase().includes(query.toLowerCase())
    );

    return (
      <ProductSearch
        query={query ?? ""}
        results={results}
        _output={text(
          `Found ${results.length} fruits matching "${query ?? "all"}"`
        )}
        _invoking="Searching..."
        _invoked="Results loaded"
        _prefersBorder={true}
      />
    );
  }
);
```

**Summary**

- **Return value**: JSX — the runtime builds `structuredContent`, `content`, and widget resource metadata.
- **`_output`**: Model-visible summary (e.g. `text(...)`).
- **`_invoking` / `_invoked`**: Status strings for hosts.
- **`_csp`**, **`_prefersBorder`**, etc.: Optional per-call metadata; see [MCP Apps](./mcp-apps) and [Content Security Policy](./content-security-policy).

You **do not** add `widget: { name: "..." }` for this pattern.

## Step 3: Widget component (plain React)

`ProductSearch` is a normal component. It reads server-supplied data with **`useWidget()`** (not React props from the parent — the host injects structured content).

```tsx
// components/ProductSearch.tsx
import { useWidget, McpUseProvider } from "mcp-use/react";

type ProductSearchProps = {
  query: string;
  results: { fruit: string; color: string }[];
};

export default function ProductSearch() {
  const { props, isPending } = useWidget<ProductSearchProps>();

  if (isPending) {
    return <div className="animate-pulse p-4">Loading...</div>;
  }

  const { query, results } = props;

  return (
    <McpUseProvider autoSize>
      <div className="p-4 rounded-lg border">
        <h3 className="font-semibold mb-2">
          Results for "{query}" ({results.length} fruits)
        </h3>
        <ul className="space-y-2">
          {results.map((r, i) => (
            <li key={i}>{r.fruit}</li>
          ))}
        </ul>
      </div>
    </McpUseProvider>
  );
}
```

<Warning>
**Loading state**: The iframe can render before the tool finishes. When `isPending` is `true`, treat `props` as not yet available.
</Warning>

<Tip>
See [Widget lifecycle](/typescript/server/widget-components/usewidget#widget-lifecycle) and [Streaming tool arguments](/typescript/server/widget-components/usewidget#streaming-tool-arguments).
</Tip>

## Step 4: Non-widget tools

Mix widget tools with tools that return `text()`, `object()`, etc.

```typescript
import { object } from "mcp-use/server";

server.tool(
  {
    name: "get-product-details",
    description: "Get detailed information about a product",
    schema: z.object({ productId: z.string() }),
  },
  async ({ productId }) =>
    object({
      id: productId,
      name: "Example Product",
      price: 29,
      description: "A great product",
    })
);
```

See [Response helpers](./response-helpers).

## Step 5: Typed `useCallTool` with `ToolRef`

Export tools you need to call from widgets:

```tsx
export const getProductDetails = server.tool(
  {
    name: "get-product-details",
    schema: z.object({ productId: z.string() }),
  },
  async ({ productId }) => object({ id: productId, name: "Widget", price: 29 })
);
```

In the widget:

```tsx
import { useCallTool } from "mcp-use/react";
import { getProductDetails } from "../index"; // server module export — adjust path

const { callTool, data, isPending } = useCallTool(getProductDetails);

<button onClick={() => callTool({ productId: "123" })}>Details</button>
```

`useCallTool` accepts a **string** tool name for untyped calls, or a **`ToolRef`** for full typing. See [useCallTool](./widget-components/usecalltool).

## Step 6: Optional batch registration

```tsx
export const tools = server.defineTools({
  echo: {
    description: "Echo input.",
    schema: z.object({ message: z.string() }),
    handler: async ({ message }) => text(message),
  },
});
```

## Best practices

**Schemas**: Use `.describe()` on Zod fields so models pick correct arguments.

**Themes**: Use `theme` from `useWidget()` for light/dark.

**Conditional UI**: If a client does not support Apps, return text:

```tsx
async (params, ctx) => {
  if (ctx.client.supportsApps()) {
    return <ProductSearch {...} _output={text("...")} />;
  }
  return text("Plain summary for this client.");
};
```

## Troubleshooting

**Widget does not appear**

- Confirm the handler returns JSX (with `@jsxImportSource mcp-use/jsx` on that file).
- Check server logs and Inspector tool errors.

**Empty props**

- Guard on `isPending` before reading `props`.

**CSP errors**

- Add domains via `_csp` on the JSX or `defineWidget` / `.config`. See [Content Security Policy](./content-security-policy).
