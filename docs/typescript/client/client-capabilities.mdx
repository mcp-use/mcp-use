---
title: "Client Capabilities"
description: "Advertise MCP client capabilities and extensions during initialize"

---

Client capabilities tell an MCP server what your client can support during the
`initialize` handshake. Use them for client-level protocol features such as
roots, sampling, elicitation, and SEP-1724 extensions.

Configure capabilities once at the top level of `MCPClient` or
`BrowserMCPClient`. They are shared by every server connection because they
describe the client implementation, not an individual server.

## Advertise MCP Apps support

Use `mcpApps()` to advertise the canonical MCP Apps extension (SEP-1865). This
lets compatible servers know your client can accept MCP Apps widget resources.

```typescript
import { MCPClient, mcpApps } from "mcp-use";

const client = new MCPClient({
  capabilities: {
    extensions: mcpApps(),
  },
  mcpServers: {
    widgets: {
      url: "http://localhost:3000/mcp",
    },
  },
});
```

`mcpApps()` returns the `extensions` fragment for the MCP Apps extension:

```typescript
{
  "io.modelcontextprotocol/ui": {
    mimeTypes: ["text/html;profile=mcp-app"]
  }
}
```

The same helper is available from the browser entry point:

```typescript
import { BrowserMCPClient, mcpApps } from "mcp-use/browser";

const client = new BrowserMCPClient({
  capabilities: {
    extensions: mcpApps(),
  },
  mcpServers: {
    widgets: {
      url: "https://example.com/mcp",
    },
  },
});
```

## Combine extensions

`capabilities.extensions` mirrors the MCP wire format, so you can spread
`mcpApps()` together with other extension declarations.

```typescript
import { MCPClient, mcpApps } from "mcp-use";

const client = new MCPClient({
  capabilities: {
    extensions: {
      ...mcpApps(),
      "com.example/preview": {
        version: "1",
      },
    },
  },
  mcpServers: {
    preview: {
      command: "npx",
      args: ["-y", "@example/preview-server"],
    },
  },
});
```

## Managed capabilities

mcp-use manages some capabilities automatically:

- `roots` is always advertised so servers can call `roots/list`.
- `sampling` is advertised when you provide `onSampling`.
- `elicitation` is advertised when you provide `onElicitation`.

```typescript
import { MCPClient } from "mcp-use";

const client = new MCPClient(
  {
    mcpServers: {
      assistant: {
        url: "http://localhost:3000/mcp",
      },
    },
  },
  {
    onSampling: async (params) => {
      // Return a CreateMessageResult for the server's sampling request.
      return {
        role: "assistant",
        content: {
          type: "text",
          text: "Sampled response",
        },
        model: "example-model",
      };
    },
  },
);
```

Avoid manually setting `roots`, `sampling`, or `elicitation` unless you are
intentionally overriding low-level behavior. mcp-use may replace those keys with
the values required by its registered handlers.

## Merge precedence

Capabilities can also be passed through the underlying MCP SDK with
`clientOptions.capabilities` on a server config. Prefer the top-level
`capabilities` field for new code.

When the same key appears in multiple places, mcp-use merges them in this order:

1. `clientOptions.capabilities`
2. Top-level `capabilities`
3. mcp-use managed capabilities (`roots`, `sampling`, `elicitation`)

Later entries win. This keeps existing SDK passthroughs working while making
the new top-level option the easiest place to declare extensions.

```typescript
import { MCPClient, mcpApps } from "mcp-use";

const client = new MCPClient({
  capabilities: {
    extensions: mcpApps(),
  },
  mcpServers: {
    widgets: {
      url: "http://localhost:3000/mcp",
      clientOptions: {
        capabilities: {
          experimental: {},
        },
      },
    },
  },
});
```

In the example above, the initialize request advertises both `experimental` and
the MCP Apps extension.
